- **核心优势 (Pros)**：

      - **并发控制 (Concurrency Control)**：为不同资源（RunSnapshot, pause/resume, answer）设计了不同粒度的并发原语（ETag, `run_revision`, `wait_set_revision`），思路正确且清晰。

      - **契约先行**：以 `openapi.yaml` 为权威信源，并用 `openapi-typescript` 验证其可用性。

  4. **高度透明的风险披露**：`已知限制与不得掩盖的风险` 部分坦诚、准确地列出了所有捷径和技术债。这种自省能力是项目长期健康发展的最重要保障。

- **缺陷与不足 (Cons)**：
  1. **外部契约与实现能力不匹配**：`openapi.yaml` 描述了全部 36 个 operation，但 M1 仅实现了 20 个。这会误导 API 消费者和客户端代码生成器，让他们以为所有功能均可用，增加了集成时的困惑。
  2. **SDK 开发者体验（DX）有待提升**：SDK 目前返回 `dict[str, Any]`，未能利用 OpenAPI 契约生成强类型的请求/响应模型。这削弱了 Python 静态类型检查的优势，增加了使用者出错的概率。
  3. **事务原子性缺失的隐性风险**：文档已明确指出“多步 Store 写入”不是原子的。虽然在单线程同步工作流下问题不突出，但这是一种脆弱的实现。任何微小的并发逻辑（如未来引入的后台任务）都可能破坏数据一致性，且问题难以复现。

- **动态维度分析 (架构可靠性分析)**：
  - **可靠性**: 在 M1 约束下（单进程、无重启），可靠性较高，因为所有操作都是同步和内存中的。但真实世界可靠性为 0，任何进程退出都意味着数据全量丢失。
  - **可观测性**: 基础具备（通过事件流和快照），但缺少结构化日志、Metrics 和 Tracing。
  - **故障隔离/恢复能力**: 基本为零。没有持久化，就没有恢复能力。`410 event cursor` 的设计是协议层面的恢复，而非系统级恢复，这一点认知清晰。
  - **安全边界**: 仅有伪装的租户隔离，无真实认证/授权。这是一个明确的、可接受的 M1 简化。

- **风险一**：**范围误解导致的不当使用** - **影响面**：**高** - **触发条件**：团队新成员或外部协作者在未详细阅读文档的情况下，将此 M1 服务部署到共享的开发或测试环境，并用于需要数据持久性的场景。 - **规避策略**：
    1. 在服务启动时打印显眼的 `WARNING` 日志，明确指出“内存存储，数据易失，请勿用于生产或持久化场景”。
    2. 在项目 `README.md` 的最顶部用警告框（Warning block）突出声明其 M1 阶段状态和限制。

- **风险二**：**API 契约的“超前承诺”** - **影响面**：中 - **触发条件**：API 消费者（无论是人工还是代码生成器）尝试调用 `openapi.yaml` 中存在但 M1 未实现的 operation，导致困惑和不必要的调试。 - **规避策略**：立即在 `openapi.yaml` 中使用扩展字段（如 `x-maturity: milestone-1` 或 `x-availability: not-implemented`）标记未实现或非 GA 的 operation，并配置 Redocly 或其他工具在文档中清晰地展示这些标记。

- **风险三**：**向持久化迁移的重构复杂度** - **影响面**：中 - **触发条件**：进入 M2 开发阶段，需要将 `store.py` 中的同步、非原子性内存操作，重构为基于持久化存储（如 PostgreSQL + Temporal）的事务性或最终一致性操作。 - **规避策略**：在 M2 规划中，将“状态管理与工作流引擎重构”作为第一个、最高优先级的技术任务。在动手前，绘制详细的状态机迁移图和数据一致性模型（例如，采用 `transactional outbox` 模式保证原子性）。
